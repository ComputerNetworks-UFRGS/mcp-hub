
# TC003 — Gateway: wrong backend URLs for internal microservices - Muito dificil de detectar e fazer o promt certo

| Field            | Value                          |
|------------------|--------------------------------|
| **Fault type**   | Code bug / Configuration       |
| **Service**      | `backend` (gateway routes)     |
| **Image tag**    | `minimal-boutique-backend:tc003`|
| **Introduced by**| Hardcoded incorrect service names |

---

## Fault description

The gateway routes in `backend/routes/gateway.py` hardcode service URLs like `http://orders:5002`, `http://products:5001`, etc. If any service name is misspelled (e.g., `http://product:5001` instead of `http://products:5001`), DNS resolution will fail and requests will be rejected. Alternatively, if a service port is wrong, connection timeouts occur.

---

## Setup

- Modify `backend/routes/gateway.py`:
```python
# Change this:
PRODUCTS_API_URL = "http://products:5001/products/"
CART_API_URL = "http://cart:5005/cart/"

# To this (typo):
PRODUCTS_API_URL = "http://prod456:5001/products/" 
CART_API_URL = "http://carxyz:5005/cart/"

```
- Rebuild and redeploy backend:
```bash
# In your build pipeline
docker build -t minimal-boutique-backend:tc003 -f Dockerfile.backend .
docker tag minimal-boutique-backend:tc003 igormsilva/minimal-boutique-backend 
docker push igormsilva/minimal-boutique-backend 

# In the cluster:
kubectl set image deployment/backend backend=igormsilva/minimal-boutique-backend:tc003 -n minimal-boutique

# Follow the status
kubectl rollout status deployment/backend -n minimal-boutique
# Wait until see
deployment "backend" successfully rolled out
```
- Generate load:
```bash
kubectl port-forward svc/frontend 5173:5173 -n minimal-boutique
# Try to view products

# OR
kubectl exec -it deployment/loadgenerator -n mcp-hub -- \
  locust \
    --host http://backend.minimal-boutique.svc.cluster.local:5000 \
    --headless \
    -u 50 \
    -r 50 \
    --run-time 300m
```

## Expected agent response

The agent should:
1. **Detect**: Gateway unable to reach products service; all product-related API calls fail
2. **Root cause**: DNS name mismatch (`product` vs `products`) or incorrect port configuration
3. **Check traces**: Identify which gateway route is failing via span attributes (`http.url`, `http.status_code`)
4. **Recommend**: Verify service DNS names in Kubernetes:
   ```bash
   kubectl get svc -n minimal-boutique
   ```
5. **Fix**: Correct the service URL and redeploy

Para o agente LLM detectar isso automaticamente
O comando crítico que resolve tudo em uma linha é:
```bash

  kubectl exec -n minimal-boutique deployment/backend -c backend -- \
    grep -rn "http://[a-z]*:[0-9]*/\|PRODUCTS_API_URL\|CART_API_URL\|ORDERS_API_URL" \
    /app --include="*.py" | grep -v ".pyc"
```
Combinado com:
```bash
# Extrair todos os hostnames usados no código e comparar com services reais

  kubectl exec -n minimal-boutique deployment/backend -c backend -- \
    grep -rh "http://[a-z]*:[0-9]*" /app --include="*.py" | \
    grep -oP "http://\K[a-z]+" | sort -u > /tmp/code_hosts.txt

  kubectl get svc -n minimal-boutique -o jsonpath='{.items[*].metadata.name}' | \
    tr ' ' '\n' | sort > /tmp/k8s_services.txt

  # Diff entre hostnames no código vs services existentes
  diff /tmp/k8s_services.txt /tmp/code_hosts.txt
```

Isso mostraria product como hostname no código que não existe como service — conclusão automática da causa raiz.
