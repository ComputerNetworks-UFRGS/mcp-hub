# TC001 — Checkout: CPU starvation due to expensive fraud detection on high-value orders (>R$500)

| Field            | Value                          |
|------------------|--------------------------------|
| **Fault type**   | Code design flaw / Synchronous blocking operation |
| **Service**      | `checkout`                     |
| **Image tag**    | `minimal-boutique-checkout:TC001`|
| **Introduced by**| Unoptimized fraud detection algorithm triggered on high-value orders |

---

## Fault description

In [backend/checkout/routes/checkout.py](backend/checkout/routes/checkout.py#L50), when the order total exceeds R$500, a **CPU-intensive fraud detection algorithm** is triggered:

```python
# In the checkout process (after total calculation):
if total > 500:  # HIGH-VALUE ORDER THRESHOLD
    # Expensive synchronous fraud detection
    fraud_risk = detect_fraud(user_id, order_items_payload, total)
    if fraud_risk > 0.8:
        return jsonify({"error": "Order blocked: High fraud risk detected"}), 403
```

The `detect_fraud()` function performs **computationally expensive operations**:
- Brute-force validation of user patterns (nested loops over transaction history)
- Unindexed database queries without pagination
- Regex pattern matching on large strings
- No caching of results

When a single user places a high-value order (>R$500), the checkout service:
- **Blocks the HTTP request thread** for 30-60 seconds (CPU-bound operation)
- **Consumes 90-100% CPU** during fraud detection
- **Times out subsequent requests** to checkout (thread pool exhaustion)
- **Cascades to frontend**: Users see long loading spinners, then timeouts

The issue is **only visible under specific conditions**:
- Users with orders >R$500 experience extreme latency
- Users with orders ≤R$500 experience normal latency
- The correlation is **easily visible in Prometheus** when filtering by order total
- **Jaeger traces show very long spans** for high-value checkouts with detailed fraud_detection child spans

## Expected agent response

The agent should be able to identify:

1. **Pattern Detection**: 
   - Request latency shows **bimodal distribution** (two distinct peaks: fast and very slow)
   - Prometheus time-series: `http_request_duration_seconds` shows correlation between `order.total` and latency
   - PromQL query: `histogram_quantile(0.95, http_request_duration_seconds_bucket{endpoint="/checkout"}) > 40` correlates with orders where total > 500

2. **Root Cause Analysis**:
   - Query Jaeger for checkout traces; filter by `order.total > 500`
   - Compare trace duration: high-value orders have 50x longer spans
   - Identify `fraud_detection` span as the bottleneck (45+ seconds)
   - Check span attributes: `fraud_detection` only exists for orders > R$500

3. **CPU Correlation**:
   - Prometheus: `rate(process_cpu_seconds_total{pod="checkout-*"}[1m]) * 1000` matches high-value order arrival times
   - CPU spikes are **synchronous** with checkout requests for orders > R$500
   - CPU usage is **100% for 45 seconds**, blocking the entire service thread

4. **Impact Assessment**:
   - Subsequent checkout requests timeout while fraud detection runs
   - Frontend request queue builds up
   - Cascading failure if multiple high-value orders arrive simultaneously

5. **Steps to Investigate**:
   ```bash
   # List checkout spans from Jaeger REST API
   curl 'http://jaeger:16686/api/traces?service=checkout' \
     | jq '.data[] | select(.spans[] | select(.operationName == "fraud_detection"))'
   
   # Compare latencies by order value
   # Query spans with attributes: find correlation between order.total and span duration
   ```

6. **Recommendation**:
   - **Asynchronous processing**: Move fraud detection to a background job queue (Celery, Redis Queue)
   - **Caching**: Cache fraud detection results (user patterns don't change every second)
   - **Timeout + Fallback**: Set timeout on fraud detection; if > 5s, allow order with manual review flag
   - **Index optimization**: If using database queries, add indices on (user_id, timestamp)
   - **Circuit breaker**: Use circuit breaker pattern for external fraud API calls
   - Example fix:
     ```python
     from celery import shared_task
     
     @shared_task
     def detect_fraud_async(user_id, items, order_id):
         """Run fraud detection in background; update order status when complete"""
         fraud_risk = detect_fraud(user_id, items, total)
         order = Order.query.get(order_id)
         order.fraud_score = fraud_risk
         db.session.commit()
     
     # In checkout route:
     order = create_pending_order(...)
     if total > 500:
         detect_fraud_async.delay(user_id, items, order.id)
     return jsonify(order), 201
     ```

7. **Validation**:
   - Rerun checkout with high-value orders after fix
   - Latency should be < 500ms regardless of order value
   - CPU should return to baseline quickly
   - Fraud detection should complete asynchronously without blocking requests
