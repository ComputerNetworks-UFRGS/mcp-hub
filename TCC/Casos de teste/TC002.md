
# TC002 — Products service: resource starvation due to misconfigured CPU/Memory limits

| Field            | Value                          |
|------------------|--------------------------------|
| **Fault type**   | Configuration / Resource Constraints |
| **Service**      | `products`                     |
| **Image tag**    | `minimal-boutique-products:tc004`|
| **Introduced by**| Overly restrictive resource limits in deployment manifest |

---

## Fault description

The `products` service is deployed with extremely low resource limits that cause **CPU throttling** and **memory pressure**:

```yaml
resources:
  requests:
    memory: "32Mi"      # Too low! (normal: 256Mi)
    cpu: "10m"          # Too low! (normal: 100m)
  limits:
    memory: "64Mi"      # Severe constraint
    cpu: "50m"          # Heavy throttling
```

Under normal traffic, the pod:
- Cannot allocate enough memory for request buffering → OOMKilled or crashes
- Gets CPU throttled → requests become very slow
- Garbage collection pauses become extreme (freezing threads for seconds)
- Cannot process even moderate load

This manifests as:
- **Sporadic 503 errors** (service temporarily unresponsive)
- **Latency spikes to 10-30 seconds** for normal requests
- **Repeated pod crashes and restarts** (CrashLoopBackOff)
- **Metrics show CPU at 100% utilization constantly**

---

## Setup

**Update the products deployment with severe resource constraints:**

```bash
kubectl set resources deployment/products \
  -n minimal-boutique \
  --requests=cpu=10m,memory=32Mi \
  --limits=cpu=50m,memory=64Mi

# Verify the change
kubectl get deployment products -n minimal-boutique -o yaml | grep -A 10 "resources:"
```

**Generate moderate load to trigger resource exhaustion:**

```bash
kubectl exec -it deployment/loadgenerator -n mcp-hub -- \
  locust \
    --host http://backend.minimal-boutique.svc.cluster.local:5000 \
    --headless \
    -u 20 \
    -r 2 \
    --run-time 10m
```

## Initial prompts for the agent

"The products service in minimal-boutique keeps crashing and requests are timing out intermittently. When the service is up, it's extremely slow. Can you investigate what's causing this degradation?"


## Expected agent response

The agent should:
1. **Detect**: Service crashes frequently with high latency spikes between crashes
2. **Pod health check**: Run `kubectl describe pod` to see OOMKilled/CrashLoopBackOff events
3. **Resource inspection**: Extract resource limits from deployment YAML:
   ```bash
   kubectl get deployment products -o yaml | grep -A 10 "resources:"
   ```
4. **Metrics analysis**: 
   - CPU at 100% constantly (throttled at 50m limit)
   - Memory approaching or exceeding 64Mi limit
   - Correlation: crashes occur exactly when memory exceeds limit
5. **Timeline analysis**: Look for pattern of crashes every 2-5 minutes as pod exhausts memory
6. **Root cause**: "Resource limits are too restrictive for normal operation"
7. **Recommend**:
   - Increase `limits.cpu` to at least 200m
   - Increase `limits.memory` to at least 256Mi
   - Add monitoring alerts for CPU throttling and memory pressure
   - Profile the application to determine appropriate minimums
8. **Validation**:
   - After fix, pod should run stably
   - Latency should return to normal (< 200ms)
   - No more CrashLoopBackOff events

## Teardown

**Restore normal resource limits:**

```bash
kubectl set resources deployment/products \
  -n minimal-boutique \
  --requests=cpu=100m,memory=256Mi \
  --limits=cpu=500m,memory=512Mi

# Verify restoration
kubectl get deployment products -n minimal-boutique -o yaml | grep -A 10 "resources:"

# Check pod recovers
kubectl rollout status deployment/products -n minimal-boutique
```
